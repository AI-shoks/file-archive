# file-archive

Личный органайзер учебных материалов. Слой 1: загрузка файлов + ручная метка
предмета + хранение по папкам + поиск по имени файла.

## Скоуп Слоя 1 (НЕ расширять без явного запроса)

- БЕЗ базы данных
- БЕЗ авторизации/аутентификации
- БЕЗ RAG, БЕЗ AI, БЕЗ векторных индексов
- Хранилище — обычные папки на диске, по одной на предмет
- Папки-предметов создаются ВРУЧНУЮ заранее. `save_file` только пишет
  в существующие папки, НЕ создаёт их на лету.

## Архитектурные решения (НЕ откатывать без явного запроса)

Эти решения приняты осознанно. Если правишь соседний код — не трогай их.

### Валидация subject

- ТОЛЬКО `if not subject or subject != Path(subject).name: raise ValueError(...)`
- НЕ использовать `is_relative_to()` / `.resolve()` для проверки subject.
- Причина: папки создаются вручную заранее, subject обязан быть ОДНИМ
  именем папки (не путём). `subject != Path(subject).name` отсекает
  `"../x"`, `"a/b"`, абсолютные пути — после неё subject физически
  не может выйти за `ROOT_DIR`, поэтому `is_relative_to` был бы мёртвым кодом.

### Валидация filename

- `safe_name = Path(filename).name` — извлекает безопасное имя, отрезая
  любой путь/`../`. Пустой результат → ValueError.
- Отличие от subject: для filename `.name` ИЗВЛЕКАЕТ имя из возможного пути;
  для subject `.name` используется как ПРОВЕРКА, что путь и есть одно имя.

### Ответ эндпоинтов

- НИКОГДА не возвращать абсолютный путь файла.
- Только `{"subject": subject, "filename": Path(saved_path).name}`.
- Причина: абсолютный путь раскрывает ОС, имя пользователя системы
  и структуру ФС — пассивная утечка для атакующего. subject и filename
  клиент прислал сам, новой информации о сервере не раскрываем.

### Исключения и HTTP-коды

- `SubjectNotFoundError`     → 404 (предмет/папка не существует)
- `ArchiveFileNotFoundError` → 404 (файл в предмете не найден)
- `FileTooLargeError`        → 413 (файл превышает лимит)
- `ValueError`               → 400 (невалидный запрос: subject, имя файла)
- Все кастомные исключения наследуются от `Exception`, НЕ от `ValueError`.
  Причина: будь они наследниками `ValueError`, ветка `except ValueError`
  перехватила бы их первой и 404/413 никогда не сработали бы.
- Порядок `except` в эндпоинте: специфичные (`SubjectNotFoundError`,
  `FileTooLargeError`) ВЫШЕ широкого `except ValueError`. Python берёт
  первый подходящий `except` — широкий, стоя первым, поглотил бы остальные.

### Коллизия имён файлов (атомарно)

- При загрузке файла с уже существующим именем — НЕ перезаписывать молча.
  `_write_unique` перебирает `safe_name`, `stem(1).suffix`, `stem(2).suffix`, …
  и для каждого пытается СОЗДАТЬ файл эксклюзивно: `open(..., "xb")`
  (= `O_CREAT|O_EXCL`). `save_file` возвращает фактический путь, эндпоинт
  отдаёт итоговое имя клиенту.
- НЕ возвращать к схеме «проверить `exists()`, потом `write_bytes()`».
  Причина: между проверкой и записью есть окно (TOCTOU), а `save_file`
  уводится в пул потоков (`run_in_threadpool`, main.py) — две параллельные
  заливки одного имени реально конкурируют и одна затёрла бы другую.
  Эксклюзивное создание делает «занять имя» и «начать запись» одной
  атомарной операцией: ровно один поток создаёт файл, остальные получают
  `FileExistsError` и берут следующее имя.
- Причина инварианта в целом: тихая перезапись = потеря данных без
  предупреждения.

### Запись не блокирует event loop

- Эндпоинт `upload_file` остаётся `async`, но синхронную запись на диск
  (`storage.save_file` → `write_bytes`) вызывает через
  `await run_in_threadpool(...)`.
- Причина: прямой синхронный `write_bytes` в `async def` блокирует весь
  event loop на время записи. storage по API не меняем — запись остаётся
  синхронной, в пул потоков её уводит транспортный слой (main.py).

### Хелперы валидации (DRY)

- `_validate_subject(subject, *, context=...)` и `_safe_filename(filename)`
  в storage.py — единая точка для проверок, ранее продублированных в
  save_file/list_files/get_file_path/seed_subjects. ФОРМА проверки subject
  не изменилась (см. раздел «Валидация subject»), хелпер лишь убирает дубли.
  `context=` меняет текст ошибки для SEED_SUBJECTS, не размножая проверку.

### Лимит размера файла

- `MAX_FILE_SIZE_BYTES` — в БАЙТАХ (`len(content)` возвращает байты).
- Известное ограничение Слоя 1: проверка идёт ПОСЛЕ `await file.read()`,
  то есть файл уже целиком в RAM. Эта проверка защищает ДИСК от записи
  больших файлов, но НЕ от OOM. Полная защита от OOM = лимит на уровне
  ASGI/Render до буферизации ИЛИ потоковое чтение по чанкам. Вне скоупа Слоя 1.

### Защита записи (Слой 2)

- `POST /upload/file` защищён ключом `UPLOAD_API_KEY` (env) через заголовок
  `X-API-Key`; зависимости эндпоинта: `enforce_upload_rate_limit`,
  `require_upload_key`.
- **Fail-closed:** ключ не задан → загрузка возвращает 503, НЕ открыта.
  Причина: «secure by default», забытый на проде ключ не должен молча
  оставлять запись публичной. НЕ менять на fail-open без явного запроса.
- Коды: ключ не настроен → 503 (это не вина клиента), неверный/нет ключа
  → 401. Сравнение — `secrets.compare_digest` (постоянное время).
- Ключ НЕ зашивать во фронтенд (`static/` публичен): UI спрашивает его у
  пользователя и хранит в его `localStorage`, шлёт заголовком.
- Rate-limit `UPLOAD_RATE_LIMIT`/60c — in-memory per-IP, стоит ВЫШЕ проверки
  ключа (чтобы тормозить перебор). IP из `X-Forwarded-For` (прокси Render).
  Это defense-in-depth, не аутентификация (заголовок подделываем).
